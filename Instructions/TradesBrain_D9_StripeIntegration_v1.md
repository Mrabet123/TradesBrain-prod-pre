**2.2 Checkout Flow --- Mobile App to Stripe**

The complete flow from worker tapping Subscribe to subscription activated:

|          |               |                                                                           |                                                                |
|----------|---------------|---------------------------------------------------------------------------|----------------------------------------------------------------|
| **Step** | **Actor**     | **Action**                                                                | **Result**                                                     |
| 1        | Mobile app    | Worker selects plan and billing cycle --- taps Subscribe                  | KYC check triggered (kyc-status-check Edge Function)           |
| 2        | Edge Function | kyc-status-check verifies both documents are verified                     | Returns can_subscribe: true or blocks with KYC status          |
| 3        | Mobile app    | Calls stripe-create-checkout Edge Function with plan_type + billing_cycle | Authorization header (user JWT) sent                           |
| 4        | Edge Function | Creates or retrieves Stripe Customer for user                             | Upserts stripe_customer_id to users table                      |
| 5        | Edge Function | Creates Stripe Subscription with payment_behavior: default_incomplete     | Returns latest_invoice.payment_intent.client_secret            |
| 6        | Edge Function | Creates EphemeralKey for customer                                         | Returns ephemeral_key.secret                                   |
| 7        | Edge Function | Returns to mobile app                                                     | { client_secret, subscription_id, customer_id, ephemeral_key } |
| 8        | Mobile app    | Initialises Stripe PaymentSheet with returned values                      | initPaymentSheet() called                                      |
| 9        | Mobile app    | Presents Stripe PaymentSheet to worker                                    | presentPaymentSheet() called                                   |
| 10       | Worker        | Enters card / uses Apple Pay / Google Pay --- confirms payment            | Stripe processes payment                                       |
| 11       | Mobile app    | PaymentSheet returns success                                              | Optimistic subscription_status = active granted immediately    |
| 12       | Stripe        | Sends webhook: invoice.payment_succeeded                                  | handle-stripe-webhook Edge Function processes                  |
| 13       | Edge Function | Updates users table: subscription_status = active, plan_type, end_date    | Also creates subscriptions and billing_history records         |
| 14       | Mobile app    | On next app open                                                          | Subscription status confirmed from database --- no discrepancy |

**3.4 Webhook Security --- Signature Verification**

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>
<tbody>
<tr class="odd">
<td><p>CRITICAL: Every webhook request MUST verify the Stripe-Signature header before processing ANYTHING.</p>
<p>The signature verification code is in handle-stripe-webhook/index.ts and kyc-webhook/index.ts.</p>
<p>If verification fails: return HTTP 400 immediately — do not process.</p>
<p>If verification passes but processing fails: return HTTP 200 — log the error — do not let Stripe retry.</p>
<p>Stripe retries failed webhooks (non-200 responses) up to 3 days with exponential backoff.</p>
<p>Returning 200 on processing errors prevents duplicate processing and ghost subscriptions.</p></td>
</tr>
</tbody>
</table>

**3.5 Idempotency --- Handling Duplicate Webhook Events**

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>
<tbody>
<tr class="odd">
<td><p>Stripe may deliver the same event multiple times. All webhook handlers must be idempotent.</p>
<p>handle-stripe-webhook: Uses upsert on subscriptions table (onConflict: stripe_subscription_id) and INSERT with UNIQUE constraint on billing_history (stripe_invoice_id). Duplicate events result in no-ops — safe.</p>
<p>kyc-webhook: Updates are idempotent — setting status=verified twice has no effect.</p>
<p>General rule: Every database operation in a webhook handler must be safe to run multiple times with the same input.</p></td>
</tr>

</table>

**5. SUBSCRIPTION LIFECYCLE**

**5.1 Subscription Status Mapping**

|                    |                        |                          |                                                                                             |
|--------------------|------------------------|--------------------------|---------------------------------------------------------------------------------------------|
| **Stripe Status**  | **TradesBrain Status** | **Access Level**         | **Notes**                                                                                   |
| active             | active                 | Full access              | Normal operating state                                                                      |
| past_due           | active                 | Full access              | Grace period --- Stripe retrying payment. TradesBrain maintains access during retry window. |
| canceled           | cancelled              | Read-only until end_date | Access maintained until subscription_end_date. Set when worker cancels.                     |
| unpaid             | expired                | Read-only only           | Payment failed after all Stripe retries exhausted.                                          |
| incomplete         | trial                  | Limited --- trial active | Initial state before first payment confirmed.                                               |
| incomplete_expired | expired                | Read-only only           | Initial payment never confirmed within 23 hours.                                            |

<td><p>INITIAL STATE: Worker creates account → subscription_status = trial, trial_queries_remaining = 10</p>

<p>NOTE: TradesBrain does NOT set cancel_at_period_end = true on the Stripe subscription. Instead, TradesBrain uses Stripe cancel_at (specific future timestamp) to cancel at period end. <p>TEAM BILLING EXAMPLES (monthly):</p>
<p>Owner + 2 members (base 3 seats): $260/month</p>
<p>Owner + 3 members (1 additional seat): $260 + $89 = $349/month</p>
<p>Owner + 5 members (3 additional seats): $260 + ($89 × 3) = $527/month</p>
<p>Owner + 10 members (8 additional seats): $260 + ($89 × 8) = $972/month</p>
<p>Stripe subscription structure for owner + 5 members:</p>
<p>Item 1: TradesBrain Team Base — qty 1 — $260/month</p>
<p>Item 2: TradesBrain Team Seat — qty 3 — $89/month × 3 = $267/month</p>
<p>Total: $527/month</p></td>

**6. STRIPE IDENTITY --- KYC INTEGRATION**

**6.1 How Stripe Identity Is Used in TradesBrain**

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>

<tr class="odd">

<p>2. Trade license proof (photo of license document)</p>
<p>Both documents must be verified before the Stripe Payment Sheet can be opened.</p>
<p>KYC sessions are created server-side (in create-team-member Edge Function or mobile app setup).</p>
<p>The verification result arrives via webhook to the kyc-webhook Edge Function.</p>

<p>// Called server-side only — never from mobile app directly</p>
<p>const session = await stripe.identity.verificationSessions.create({</p>
<p>type: "document",</p>
<p>metadata: {</p>
<p>user_id: supabaseUserId, // REQUIRED — used by kyc-webhook to update correct user</p>

**6.3 KYC Status Flow**

|              |                                |                          |                                                               |
|--------------|--------------------------------|--------------------------|---------------------------------------------------------------|
| **Status**   | **Meaning**                    | **Worker Can Subscribe** | **Action**                                                    |
| not_uploaded | Document not submitted yet     | No                       | Prompt worker to upload in Settings → Profile                 |
| pending      | Stripe Identity processing     | No                       | Show pending notice. Push notification when resolved.         |
| verified     | Document verified              | Yes (if both verified)   | Lock document --- read-only forever                           |
| rejected     | Document could not be verified | No                       | Show rejection reason. Re-upload button. New session created. |

<colgroup>
<col style="width: 100%" />
</colgroup>
<tbody>
<tr class="odd">

**8. ERROR HANDLING**

**8.1 Checkout Errors --- Mobile App**

|                         |                                      |                                                             |                                                                  |
|-------------------------|--------------------------------------|-------------------------------------------------------------|------------------------------------------------------------------|
| **Error Code**          | **Cause**                            | **Mobile App Response**                                     | **User Sees**                                                    |
| kyc_required            | KYC not verified before checkout     | Do not open PaymentSheet. Show KYC status screen.           | KYC pending or rejected notice with specific document and action |

**8.2 Webhook Processing Errors**

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>
<tbody>
<tr class="odd">
<td><p>RULE: All webhook processing errors return HTTP 200 to prevent Stripe from retrying.</p>
<p>Errors are logged to Supabase logs table for investigation.</p>
<p>COMMON ERRORS AND HANDLING:</p>
<p>User not found for customer ID:</p>
<p>→ Log warning. Return 200. This can happen if a test subscription was created for a deleted user.</p>
<p>Database update fails:</p>
<p>→ Log error with full event details. Return 200.</p>
<p>→ Implement reconciliation job (Year 2) to catch missed webhooks.</p>
<p>Duplicate webhook delivery:</p>
<p>→ Upsert and UNIQUE constraints make all operations idempotent. Safe to process twice.</p>
<p>Unknown event type:</p>
<p>→ Log as info. Return 200. No action needed.</p></td>
</tr>
</tbody>

**8.3 Subscription Recovery --- Payment Failure**

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>
<tbody>
<tr class="odd">
<td><p>When invoice.payment_failed fires:</p>
<p>1. TradesBrain sends push notification: "Payment failed — update your payment method"</p>
<p>2. Stripe status = past_due → TradesBrain keeps status = active (grace period)</p>
<p>3. Stripe automatically retries payment (Smart Retries enabled in Stripe settings)</p>
<p>4. If retries succeed: invoice.payment_succeeded fires → status confirmed</p>
<p>5. If all retries fail: Stripe cancels subscription → customer.subscription.deleted fires</p>
<p>6. handle-stripe-webhook sets status = expired → features blocked</p>
<p>NOTE: Enable Smart Retries in Stripe Dashboard → Settings → Billing → Automatic collection.</p></td>
</tr>
</tbody>
</table>
